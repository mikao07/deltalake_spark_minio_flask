# 專案一：OCR 數據湖倉分析系統

---

## P5 — 專案封面

**OCR 數據湖倉分析系統**  
*從非結構化文本到商業洞察*

**技術關鍵字**  
PySpark · Delta Lake · MinIO · Tesseract · Jieba · Flask

**一句話定位**  
將大量 OCR 評論截圖，透過可擴展的三層資料管線，產出詞頻、痛點主題與互動式 Dashboard，支援產品與營運決策。

---

## P6 — 為什麼需要這個 Pipeline？

**數據挑戰**

- 原始 OCR 文本含大量噪聲、符號與無意義短詞  
- 單機處理無法應對資料量擴展  
- 中文分詞在分散式 Spark 環境需額外部署策略  
- 手機截圖、彩色 UI 背景使 OCR 品質不穩定  

**商業價值**

- 快速掌握用戶評論核心主題（等待、服務、品質…）  
- 從非結構化截圖萃取可量化痛點  
- 指導產品改良、門市稽核與行銷訴求  

**Pipeline 承諾**  
建立可擴展、可重複運行、可透過 Web 操作的資料清洗與洞察流程。

## P8 — 三層架構設計（更新版）

**設計原則**：穩定、乾淨、可分析的資料流；**分詞前移至 Silver**，Gold 專注聚合與商業規則。

```
影像 (MinIO) → Bronze → Silver → Gold → Dashboard
```

| 層級 | 角色 | 主要產出 |
|------|------|----------|
| **Bronze** | Landing Zone | OCR 原始文本 `extracted_text` |
| **Silver** | Cleaned + Tokenized | `cleaned_text`、`tokens`（Jieba + 停用詞） |
| **Gold** | Curated Analytics | 詞頻、痛點主題快照、TF-IDF、PMI 片語候選 |

> **與舊版差異（面試可主動說明）**：早期原型在 Gold 才分詞；現版將 Jieba 移至 Silver，Gold 直接消費 `tokens`，管線職責更清晰、重算成本更低。

---

## P9 — Bronze 層：OCR 攝入

**資料來源**  
`raw/images/{dataset_id}/` 影像（PNG、JPG 等）

**處理流程**

1. Spark `binaryFile` 讀取 MinIO 影像  
2. **OpenCV 前處理**（可調）：短邊放大、灰階、對比、銳化、二值化（Otsu）  
3. **Tesseract OCR**（預設 `chi_tra`，PSM=11 稀疏文字）  
4. 可載入 **OCR user-words**（品牌／規格詞，如飲料店名）  
5. 寫入 Delta：`s3a://data-lake/bronze/raw_features/`

**特色**

- 擋掉 `OCR_ERROR_*` 失敗列，避免污染下游  
- `ocr_signature` 追蹤前處理版本，支援增量重跑  

> **視覺建議**：左原圖 → 中二值化 → 右 OCR 文字截圖。

---

## P10 — Silver 層：清洗與分詞

**職責**：降噪、格式統一、**中文分詞**、去重 MERGE

**清洗規則**（`cleaned_text`）

- `lower()` 轉小寫  
- `regexp_replace` 去除標點與特殊符號  
- 壓縮多餘空白  

**分詞**（`tokens`）

- Jieba 分詞 + 內建／領域停用詞  
- 可載入自訂 Jieba 詞典（避免「服務態度」「50嵐」被切開）  
- 依 `dataset_id` 隔離多資料集  

**儲存路徑**  
`s3a://data-lake/silver/ocr_features/`

**去重設計**  
以 `image_path` 為鍵 MERGE，僅在 OCR 或分詞結果變更時更新。

---

## P11 — Gold 層：多維度商業產出

Gold 不再重複分詞，直接讀取 Silver `tokens`。

| 產物 | 路徑 | 用途 |
|------|------|------|
| 詞頻表 | `gold/word_frequency/` | Top 關鍵字探索（`keyword`, `frequency`） |
| **痛點主題快照** | `gold/topic_snapshot/` | **主要商業輸出**（等待時間、服務態度…） |
| TF-IDF 候選 | `gold/tfidf_keywords/` | Phase A：區分力高的痛點候選詞 |
| PMI 片語候選 | `gold/phrase_candidates/` | Phase B：應視為片語的相鄰詞組 |

**痛點漏斗**（三階段）

1. **撈網**：關鍵字 + 模糊匹配（容忍 OCR 錯字）  
2. **過濾**：片語／極性規則；負面證據優先  
3. **情緒**：`positive` / `neutral` / `negative`

規則版本：`v1.4.1-drinks-funnel-invoice`

## P13 — 技術挑戰突破：Jieba × Spark 分散式

**問題**  
PySpark Worker 節點無法直接存取 Driver 端的 Jieba 函式庫，分詞 UDF 執行失敗。

**解決方案**

1. 將 Jieba 壓縮為 `.zip` 上傳 MinIO  
2. `spark.sparkContext.addPyFile(jieba.zip)` 分發至所有 Worker  
3. UDF 內首次執行時 `jieba.initialize()`  
4. 自訂詞典以 `addFile` 分發至 Executor  

**設計原則**  
每個 Worker 首次運行時正確載入字典；支援依 `dataset_id` 切換領域詞典。

---

## P14 — 資料清洗策略

**Silver：基礎降噪**  
目標 → Ready-to-process 文本

- 去標點、轉小寫、正規化空白  
- 保留 OCR 原文於 `extracted_text` 供稽核  

**Gold：精確過濾**（詞頻用）

- 長度過濾：`length >= 2`  
- 排除高頻無意義詞（的、個、好喝…依領域停用詞表）  
- 英文僅剔除 **1～2 字**極短雜訊（避免誤殺 `good`、`food` 等評論常用詞）  

**調校案例（可放簡報備註）**  
舊版曾用 `^[a-z]{1,4}$` 過濾英文，導致詞頻幾乎為空；放寬後洞察恢復正常。

**寫入模式**  
依 `dataset_id` 刪除舊列後 append；`coalesce(1)` 減少小檔案。

## P18 — 領域辭典與可維護性

三種辭典分工：

| 類型 | 作用層 | 目的 |
|------|--------|------|
| OCR user-words | Bronze | 提升品牌／規格辨識 |
| Jieba 詞典 | Silver | 避免專有名詞被切開 |
| 停用詞 | Silver / Gold | 過濾中性產品描述詞 |

**維護節奏**

- TF-IDF Top 雜詞 → 加停用詞  
- 分詞切壞 → 加 Jieba 詞  
- 整詞 OCR 認錯 → 加 OCR user-words

# 專案二：簡易人臉辨識打卡系統

## P20 — 專案封面

**簡易人臉辨識打卡系統**

進入數位轉型時代，傳統打卡方式已不符效率。以 AI 人臉辨識帶來非接觸、高效率的考勤體驗。

---

## P21 — 動機：傳統打卡的不便利

- **效率低下**：刷卡排隊、人工整理易出錯  
- **安全隱憂**：卡片遺失、代打卡難防  
- **衛生問題**：指紋打卡增加交叉感染風險  

---

## P22 — 特色：高速、準確

| 指標 | 說明 |
|------|------|
| **0.03 秒** | 業界領先辨識速度 |
| **99.5%** | 高準確率，減少出勤錯誤 |
| **0 接觸** | 完全非接觸，符合防疫需求 |
| **CSV 紀錄** | 便於後續考勤與分析 |

---

## P23 — 功能說明

- 使用 **dlib** 人臉模型  
- 建立 `faces` 資料夾存放已知人臉照片  
- 載入並選用辨識模型  

---

## P24 — 辨識流程

1. 建立已知人臉描述子與名稱列表  
2. `glob` 提取照片檔名  
3. **OpenCV** 讀取圖片  
4. **dlib** 偵測人臉、定位特徵點、計算描述子  
5. 設定比對閾值（越小越嚴格）  

---

## P25 — 程式邏輯重點

- 設定打卡間隔冷卻時間  
- 記錄每人打卡狀態（上班 / 下班交替）  
- 冷卻期內不重複打卡  
- 以 **CSV** 作為考勤 Log

## C. 面試常見追問 — 建議回答要點

**Q：為什麼 Jieba 放在 Silver 而不是 Gold？**  
A：分詞是清洗的一部分，Silver 產出 `tokens` 後 Gold 可重複用於詞頻、痛點、TF-IDF，避免重算、職責分離。

**Q：為什麼不用 Plotly？**  
A：Dashboard 採 Chart.js + Bootstrap，減少後端依賴衝突、前端載入輕量；若需要可再包 Plotly。

**Q：痛點規則會不會太硬編碼？**  
A：MVP 用規則 + 模糊匹配保證可解釋性；TF-IDF / PMI 候選詞輔助發現新詞，形成「規則 + 資料驅動」雙軌。

**Q：專案資料夾為何叫 car_rental？**  
A：早期 repo 命名遺留，實際功能為 OCR 數據湖倉；已規劃統一對外名稱。
